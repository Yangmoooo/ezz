# ezz v3 设计基线

状态：已确认

确认日期：2026-07-21

本文档固定 ezz v3 的产品范围、行为契约、架构约束和发布标准。除非后续通过明确决策修订本文档，否则实现应以此为准。

文中的关键字含义如下：

- **必须**：v3.0 发布所需的硬性要求。
- **不得**：明确排除或禁止的行为。
- **可以**：允许实现，但不是 v3.0 的必要条件。

## 1. 产品定义

ezz 是一个无主窗口的桌面解压工具。用户从系统文件管理器打开一个或多个文件，ezz 使用随应用发布的 7-Zip 引擎完成格式识别、密码尝试、解压、目录整理和原归档清理。

v3 的核心承诺是：

- 一次操作完成绝大多数常见归档文件的解压。
- 支持 Steganographier 生成的 MP4/MKV 隐写文件。
- 支持仅修改文件扩展名、但仍可被 7-Zip 识别的归档。
- 不覆盖现有文件，不提交部分结果，不永久删除原归档。
- Windows 和 macOS 提供一致的核心解压行为。

程序显示名、Cargo 包名和产品名统一为 `ezz`。macOS Bundle Identifier 为 `io.github.yangmoooo.ezz`。

程序内的通知、弹窗和错误信息只使用英文。README 和面向当前用户的主要文档使用中文。

## 2. 支持范围

v3.0 只支持：

- Windows 10/11，`x86_64-pc-windows-msvc`。
- macOS 11 及以上，`aarch64-apple-darwin`。

v3.0 明确不支持：

- Linux。
- Windows ARM。
- macOS Intel (`x86_64`)。
- macOS 10.x。

源码必须对其他目标给出明确的编译错误，而不是产生缺少平台符号的间接错误。

v2 的 tag、release 和历史源码必须保留，但 v3 发布后不再维护 v2，不再为其修复功能问题或更新 7-Zip。v3 不保证兼容 v2 的密码库、配置、命令行或发布结构。

## 3. 用户入口和应用生命周期

### 3.1 桌面优先

v3 是桌面程序，不再把终端调用作为正式支持的用户接口。

- 必须删除 `add` 和 `extract` 子命令。
- 必须删除对 `clap` 的依赖。
- 不提供稳定的命令行参数、帮助文本或终端输出契约。
- Windows Adapter 可以从进程参数接收 Explorer 或右键菜单传入的文件路径。
- macOS Adapter 必须从 AppKit 打开文件事件接收路径。
- 直接启动应用且没有输入文件时，必须显示原生文件选择器。
- 文件选择器必须允许用户选择任意文件，以支持修改过扩展名的归档。

### 3.2 无主窗口

v3 不提供主窗口、任务列表、进度条、暂停或取消功能。

程序只在以下情况显示界面：

- 用户直接启动应用时显示文件选择器。
- 已保存密码全部失败时显示密码弹窗。
- 发生需要用户知晓的错误时显示错误或通知。

开始处理时可以发送通知。队列处理完成后必须发送汇总通知。

### 3.3 多文件队列

v3 必须支持一次打开多个输入文件。

- 所有输入进入单进程队列。
- 队列必须严格顺序处理，不并行解压。
- 单个输入失败不得阻止后续输入。
- 处理期间再次打开的文件必须追加到现有队列。
- Windows 必须将后续进程收到的文件转发给现有实例。
- macOS 使用系统应用事件将文件交给现有实例。
- 队列清空、状态保存和通知完成后，应用必须退出，不得常驻后台。

## 4. macOS 桌面集成

macOS 必须发布标准 `.app`，并通过 `Info.plist` 注册支持的文档类型。

首发注册范围为：

- `7z`, `zip`, `rar`
- `tar`, `gz`, `tgz`
- `bz2`, `tbz`, `tbz2`
- `xz`, `txz`
- `zst`, `tzst`
- `lz`, `lzma`
- `cab`, `arj`, `lzh`, `cpio`
- `001`, `z01`
- `mp4`, `mkv`

不得把 ezz 注册成所有 `public.data` 文件的通用打开程序。不得为 `dmg`、`iso`、`exe`、`msi`、虚拟磁盘或文件系统镜像注册关联。ezz 不得在安装或首次运行时抢占系统默认打开程序。

对于未注册或修改过扩展名的文件，用户必须直接启动 ezz 后通过文件选择器选择。v3.0 不承诺 Finder 会把未注册类型通过拖放或打开文件事件交给 ezz。

MP4/MKV 关联仅提供 Steganographier 输入入口。普通视频必须在只读探测阶段被判定为不支持，不得创建输出或清理原视频。

## 5. 解压工作流

每个输入必须经过以下顺序：

1. 规范化输入路径并识别分卷集合。
2. 只读探测普通归档或特殊封装格式。
3. 在归档所在文件系统创建随机隐藏工作目录。
4. 在工作目录中准备实际归档输入。
5. 解析并尝试密码候选。
6. 使用 7-Zip 解压到工作目录。
7. 验证输出的路径和文件类型安全性。
8. 计算最终目录布局和无冲突名称。
9. 将完整结果提交到归档所在目录。
10. 根据本次任务选项将原归档及分卷移入废纸篓。
11. 清理正常流程产生的工作目录并返回结构化结果。

只有完成第 9 步后，程序才可以清理原归档。

### 5.1 事务式提交

- 7-Zip 必须只向工作目录解压，不得直接写入最终路径。
- 工作目录必须位于输入归档旁，以便最终结果在同一文件系统内重命名提交。
- 7-Zip 退出码只有 `0` 表示成功。
- 警告、CRC 错误、缺少分卷或部分解压都必须视为失败。
- 失败时不得提交部分结果，也不得清理原归档。
- 正常成功或失败时必须清理本次工作目录。
- 程序崩溃、被强制结束或系统断电后遗留的工作目录不自动扫描或删除，由用户手动处理。

v3 不额外实现压缩炸弹检测、磁盘空间监控、文件数量限制或压缩率限制。这些能力不属于 ezz 作为 7-Zip 包装器的职责。

### 5.2 输出目录布局

输出布局必须根据工作目录中的有效顶层内容计算：

- 只有一个有效顶层文件或目录时，直接提交该项。
- 有多个有效顶层项时，将它们放入以归档命名的目录。
- `__MACOSX` 和 `.DS_Store` 不参与顶层数量判断，并默认丢弃。
- 其他隐藏文件属于有效内容，不得因为不可见而丢弃。
- 最多剥离一层目录，不递归猜测或消除多层同名目录。
- 普通归档只解压一层。输出中的内层归档不得自动递归解压。
- Steganographier 是明确识别的封装格式，不受普通归档单层规则限制。

### 5.3 名称冲突

v3 不得覆盖现有文件，也不得把输出合并到现有目录。

- 单文件冲突使用 `name (1).ext`, `name (2).ext` 等名称。
- 目录冲突使用 `name (1)`, `name (2)` 等名称。
- 冲突处理作用于完整提交结果，不得依赖 7-Zip 分别重命名内部文件。
- 成功结果和通知必须报告最终实际路径。

### 5.4 路径和文件安全

- 必须拒绝会逃出工作目录的路径。
- 必须拒绝绝对路径、设备文件、FIFO 等特殊文件。
- 相对符号链接只有在解析后仍指向解压结果内部时才允许。
- Windows 无法安全创建或验证的符号链接必须导致该输入失败。
- 任何安全验证失败都必须保留原归档。

## 6. 格式处理

### 6.1 7-Zip 主引擎

7-Zip 是 v3 唯一的归档解压引擎。v3.0 不引入 `libarchive`、纯 Rust 格式库或 `unar` 作为回退引擎，也不建立公开插件系统。

普通归档的格式识别必须依赖 7-Zip 的内容探测能力，而不是依赖扩展名。扩展名只用于系统文件关联、分卷定位和输出命名。

解析归档列表时应使用稳定的技术输出模式，不得继续依赖面向人类的列表布局正则表达式。

### 6.2 特殊格式 Adapter

特殊格式在解压工作流内部形成真实 seam，至少包含：

- 普通归档 Adapter。
- Steganographier Adapter。

Steganographier Adapter 必须通过内容或内部结构确认格式，不得把所有 MP4/MKV 视为隐写文件。它必须在工作目录中生成或定位普通归档输入，不得在原视频旁创建固定名称的 `2.zip`。

未来新增特殊格式时，可以增加内部 Adapter，但不得要求桌面 Adapter、密码逻辑或目录提交逻辑了解格式细节。

### 6.3 分卷归档

v3 必须接受用户选中的任意分卷，并自动定位首卷：

- `.002` 等数字分卷回溯到 `.001`。
- `.partN.rar` 回溯到 `.part1.rar`。
- `.z01`, `.z02` 等定位对应的 `.zip`。

找不到首卷或确认存在分卷缺口时必须失败，不得清理任何分卷。解压和提交成功后，所有确认属于该归档的分卷必须作为一个集合移入废纸篓。

## 7. 密码处理

密码候选必须按以下顺序处理：

1. 空密码。
2. 最近成功使用的已保存密码。
3. 其余已保存密码。
4. 用户通过原生密码弹窗输入的密码。

已保存密码全部失败后：

- 必须显示原生密码输入弹窗。
- 输入错误时必须保留或重新显示弹窗，并允许继续尝试。
- 用户可以取消；取消表示当前输入失败，批处理继续。
- 弹窗必须包含 `Remember this password`，默认勾选。
- 弹窗必须包含 `Keep the original archive`，默认不勾选。
- 密码只有在验证和完整解压成功后才能保存。
- 保存时必须去重并更新最近使用信息。

`Keep the original archive` 只在密码弹窗出现时可选。无密码归档或已保存密码命中的归档在成功后始终执行默认清理。

v3 使用新的结构化明文密码库，不读取或迁移 v2 的 `.ezz.pw`。密码库必须原子写入，并使用当前用户可访问的严格文件权限。日志不得记录密码或完整 7-Zip 密码参数。

v3.0 不使用 macOS Keychain 或 Windows Credential Manager，也不提供密码管理窗口。

## 8. 原归档清理

- 默认清理行为是移入系统废纸篓或回收站，绝不永久删除。
- 只有解压结果完整提交后才能开始清理。
- 特殊格式原文件遵循相同规则。
- 分卷归档必须作为已确认的完整集合清理。
- 分卷识别存在不确定性时不得自动清理。
- 清理失败只产生警告，已提交的解压结果仍然算成功。
- `Keep the original archive` 被勾选时，本次输入及其分卷全部保留。

## 9. 模块设计

v3 必须围绕一个深的解压工作流模块组织。桌面调用方不应了解 7-Zip 命令、特殊中间文件、密码排序、工作目录或目录整理细节。

概念结构如下：

```text
Windows Desktop Adapter ----+
                            +--> DesktopApplication
macOS Desktop Adapter ------+          |
                                       v
                              ExtractionWorkflow
                               /       |       \
                        Input format  Password  Layout/commit
                         handlers     store        |
                               \       |       /
                                  SevenZip
```

模块职责：

- `DesktopApplication`：接收打开文件事件、维护顺序队列、调用工作流并发送通知。
- `ExtractionWorkflow`：对单个输入执行完整行为契约并返回结构化结果。
- 格式处理实现：探测并准备普通归档或 Steganographier 输入。
- 密码库实现：加载、排序、去重和原子保存密码记录。
- `SevenZip`：封装固定版本 7-Zip 的进程调用和结果解释。
- Windows/macOS Adapter：处理系统事件、原生弹窗、文件选择器、通知、数据路径和单实例行为。

`SevenZip` 目前只有一个实现，因此不得为了假设中的第二引擎扩大其接口。格式处理已有普通归档和 Steganographier 两个实现，因此该内部 seam 是必要的。

解压工作流的接口同时是主要测试表面。测试必须验证可观察结果，不得依赖内部函数调用顺序。

## 10. 数据和日志

v3 没有持久化设置文件。`Keep the original archive` 和 `Remember this password` 只属于当前密码弹窗。

持久数据位置：

```text
Windows password database:
%APPDATA%\ezz\passwords.json

Windows logs:
%LOCALAPPDATA%\ezz\logs\ezz.log

macOS password database:
~/Library/Application Support/ezz/passwords.json

macOS logs:
~/Library/Logs/ezz/ezz.log
```

密码库和日志不得放在程序目录。v3 不读取程序旁的 `.ezz.pw` 或 `ezz.log`。

日志可以包含输入路径、输出路径、格式判断、7-Zip 退出码和整理结果，但不得包含密码。日志轮换策略属于实现细节，不构成对用户的稳定接口。

## 11. 7-Zip 版本和打包

ezz 必须固定具体的 `7zz-bin` tag、资产名和 SHA-256。不得在构建时请求 `latest`。同一个 ezz tag 必须始终选择同一个 7-Zip 版本。

普通 `cargo build` 不得访问网络，也不负责组装最终桌面发布物。

仓库必须提供 Rust `xtask`：

- `cargo xtask prepare`：下载固定平台资产、校验 SHA-256 并缓存到 `target/ezz-tools/`。
- `cargo xtask package`：构建 ezz 并组装对应平台发布物。

开发运行和端到端测试使用已校验的缓存。缓存缺失时必须给出明确操作提示。

### 11.1 Windows 发布物

Windows 发布 ZIP，不提供安装器：

```text
ezz-<version>-windows-x64.zip
└── ezz-<version>-windows-x64/
    ├── ezz.exe
    ├── 7zz.exe
    └── license files
```

ezz 和 `7zz.exe` 并排放置。不得在运行时释放内嵌的 7-Zip。用户通过 Windows 打开方式或 Custom Context Menu 自行配置入口，程序不得修改注册表或抢占默认关联。

### 11.2 macOS 发布物

macOS 发布 ZIP，不提供 DMG：

```text
ezz-<version>-macos-arm64.zip
└── ezz.app/
    └── Contents/
        ├── Info.plist
        ├── MacOS/
        │   ├── ezz
        │   └── 7zz
        └── Resources/
            ├── application assets
            └── license files
```

首发使用 ad-hoc 签名。必须先签名嵌套的 `7zz`，再签名应用包。README 必须说明从网络下载后如何通过 Finder 右键打开或系统隐私与安全设置完成首次 Gatekeeper 放行。

正式 Developer ID 签名和公证可以在未来加入，但不得要求修改应用结构或解压核心。

## 12. 平台实现

- 解压核心必须使用 Rust。
- Windows 使用原生 Win32/NWG Adapter。
- macOS 使用 `objc2` / `objc2-app-kit` 实现 AppKit Adapter。
- 通知可以继续使用 `notify-rust`。
- 不引入 Swift 工程。
- 不引入 Tauri、Electron、WebView 或前端资源。

平台 Adapter 可以有不同实现，但必须向 `DesktopApplication` 提供一致的文件事件、密码询问、文件选择、通知和数据目录行为。

## 13. 测试和发布门槛

v3.0 发布前，Windows x64 和 macOS arm64 CI 都必须通过真实固定版本 7-Zip 的端到端测试。

最低测试矩阵包括：

- 普通 ZIP、7z、RAR、tar、gzip 和 xz。
- 无密码、内容加密、文件名加密和错误密码。
- 数字分卷、RAR 分卷和 ZIP 分卷。
- 从非首卷输入自动定位首卷。
- 单文件、单目录、多顶层内容和平台元数据。
- Unicode、空格和冲突名称。
- 普通 MP4/MKV 不得被误判。
- Steganographier MP4/MKV 的成功和失败路径。
- 绝对路径、路径穿越、符号链接和特殊文件。
- 7-Zip 警告、损坏归档和缺卷不得提交部分结果。
- 批处理中单个失败后继续处理。
- 成功提交后清理原归档；失败时保留原归档。
- 清理失败时结果保持成功并包含警告。

普通测试归档可以在测试时动态生成。无法由 7-Zip 生成的 RAR、分卷和 Steganographier 样本可以作为小型、无敏感内容的固定文件提交到 `tests/fixtures/`。每个固定样本必须记录生成工具、版本、密码和预期内容。

## 14. 明确不在 v3.0 范围内

- Linux 或其他 Unix 桌面支持。
- macOS Intel 和 Windows ARM。
- 正式 CLI 和子命令。
- 主窗口、设置窗口、密码管理窗口、任务列表和进度条。
- 暂停、取消和后台常驻。
- 自动更新。
- Windows 安装器或 macOS DMG。
- 正式 Apple Developer ID 签名和公证。
- Finder 快速操作或通用 `public.data` 文件关联。
- 公开特殊格式插件系统。
- 多解压引擎和失败后的引擎回退。
- 普通归档递归解压。
- 压缩炸弹、磁盘空间或文件数量监管。
- 自动清理异常中断后遗留的工作目录。
- v2 密码库、日志位置、CLI 或配置迁移。

## 15. 实施顺序

1. 建立 `tests/fixtures/` 和 v3 所需的行为样本。
2. 将核心逻辑重组为可测试的 Rust library，并定义结构化结果。
3. 删除 Linux、旧 CLI、旧构建下载和 v2 密码库实现。
4. 实现事务式 `ExtractionWorkflow`、输出布局和安全验证。
5. 实现新密码库和密码询问 seam。
6. 实现 Windows 桌面 Adapter、单实例转发和 Portable ZIP。
7. 实现 macOS AppKit Adapter、`.app`、文件关联和 ad-hoc 签名。
8. 实现固定版本 7-Zip 的 `xtask` 准备与打包流程。
9. 完成双平台 CI、端到端测试、许可证文件和 README。
10. 发布 `3.0.0`，同时将 v2 标记为停止维护。

## 16. 完成标准

只有同时满足以下条件，v3.0 才可以发布：

- 支持范围内的两个平台均能从系统文件管理器启动。
- 已注册格式、修改后缀归档和 Steganographier 样本通过测试。
- 多文件队列、密码弹窗、结果布局和清理契约全部通过端到端测试。
- 任一失败路径都不会覆盖现有内容、提交部分结果或清理唯一原归档。
